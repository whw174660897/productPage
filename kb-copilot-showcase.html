// 安装与运行

从安装包到 Obsidian 内运行

不要从命令开始，而是从前置环境和发布包开始。普通用户需要先安装 Obsidian，再下载 Knowledge Base Copilot 安装包，最后运行脚本并在 Obsidian 中启用插件。

安装前需要准备

  • Obsidian 桌面端 1.4+
    用于打开 Vault 并加载 Knowledge Base Copilot 插件。
  • Python 3.11+
    用于运行本地 kb-service 服务，Windows 安装时建议勾选 Add Python to PATH。
  • 一个 Obsidian Vault
    可以使用已有库，也可以新建文件夹作为知识库根目录。
  • 飞书功能可选依赖
    如需批量导入飞书/Lark 文档，安装 Node.js 18+ 与 npm install -g @larksuite/cli。
安装脚本会检测 Obsidian 和 lark-cli。未安装 lark-cli 不影响基础功能，只会暂时不可用飞书批量导入。

普通用户完整路径

下载发布包进入 AtomGit 发布仓库，下载 kb-copilot-v*.zip，不要只复制命令。
解压安装包解压后确认目录中包含 install.ps1、kb-service/、obsidian-plugin/ 和配置说明。
运行安装脚本Windows 使用 PowerShell 运行 install.ps1；macOS/Linux 使用 bash install.sh。
选择 Vault 并写入配置脚本会初始化 Vault 目录、安装本地服务、复制 Obsidian 插件，并提示填写 LLM API Key。
在 Obsidian 启用插件打开设置 → 社区插件 → 已安装插件，启用 Knowledge Base Copilot，然后测试连接。

为什么安装步骤要先写 Obsidian 和安装包？

因为插件必须运行在 Obsidian 桌面端里，普通用户也需要先从发布仓库下载完整 zip。只展示 install.ps1 或 bash install.sh 会让用户误以为复制命令就能完成部署。

为什么管理台会嵌入 Obsidian 标签页？

这样用户不需要离开 Obsidian，就能完成导入、浏览、搜索、问答和服务管理。插件负责入口与生命周期，React 管理台负责复杂交互。

为什么需要本地 FastAPI 服务？

检索、存储、AI 编排和文件处理更适合由独立本地服务承担。插件保持轻量，服务层负责能力扩展，二者通过本地 HTTP 通信。

飞书/Lark 导入是不是必装？

不是。lark-cli 只影响飞书/Lark 批量导入能力，不影响 URL、本地文件、微信文章、检索和问答等基础流程。需要飞书导入时再安装 Node.js 与 @larksuite/cli 即可。

和普通笔记插件有什么不同？

普通插件偏记录与展示；Knowledge Base Copilot 更像本地知识操作系统，重点是摄取、索引、治理、问答和运行时管理。
